## 1. Workflow vs Agent

**Workflow** — это когда мы жёстко прописываем цепочку шагов: сначала отправляем промпт → получаем ответ LLM → потом другой промпт с результатом → снова ответ. Каждый шаг предопределён кодом.

**Agent** — это когда LLM сама решает, когда и какой инструмент вызвать, в каком порядке, и когда остановиться.

**Правило:** по умолчанию используем workflow. Переходим к агенту только когда workflow не хватает.

---

## 2. Augmented LLM

Это усиленная LLM — база для любого агента. Состоит из трёх компонентов:

- **RAG (Retrieval)** — доступ к документам, базам данных
- **Tools** — возможность вызывать функции (поиск, API, калькулятор)
- **Memory** — запоминать историю диалога

Без этого LLM — просто болванка. С этим — основа для построения агента.

---

## 3. Пять паттернов построения агентов (из Building Effective Agents)

1. **Prompt Chaining** — шаг за шагом, результат A → вход B. Для задач, где каждый шаг важен сам по себе.
2. **Routing** — LLM классифицирует запрос и направляет в нужный модуль.
3. **Parallelization** — запуск нескольких LLM одновременно для разных задач.
4. **Orchestrator-Workers** — центральный агент (оркестратор) управляет саб-агентами, каждый отвечает за свою функцию.
5. **Evaluator-Optimizer** — одна LLM генерирует, другая оценивает и даёт фидбек.

---

## 4. Четыре контекстных примитива (Context Engineering)

1. **Write** — писать качественные инструкции для LLM: персона, задача, формат вывода, стиль.
2. **Select** — выбирать только релевантную информацию в момент запроса, не тащить весь контекст (поиск по файлам, RAG).
3. **Compress** — сжимать контекст, когда он заполняется (суммаризация, фильтрация шума).
4. **Isolate** — изолировать контексты между разными частями системы / саб-агентами.

**Sandwich pattern:** самые важные инструкции — в начале (системный промпт) и в конце (формат вывода). Переменный контекст — в середине. Модель лучше запоминает начало и конец.

## 5. Состояние индустрии агентов (State of Agent Engineering, LangChain, 2026)

Опрос 1300+ профессионалов. Ключевые цифры:

- **57.3%** организаций уже имеют агентов в продакшене
- **Качество** — главный барьер (не стоимость, как раньше)
- **89%** внедрили observability, **52%** — evaluation
- Большинство использует **несколько моделей**, а не одну
- **OpenAI** — лидер, но Gemini, Claude и open-source тоже активно используются

### Топ-3 агентов в повседневной работе:
1. **Coding agents** (Claude Code, Cursor, Copilot, Windsurf)
2. **Research & Deep Research agents** (ChatGPT, Claude, Gemini, Perplexity)
3. **Custom agents** (на LangChain, LangGraph — под свои задачи)

### Топ use case для агентов:
- **Customer support** (26.5%) — агенты прямо перед клиентами
- Research & data analysis (24.4%)
- Internal workflow automation (18%)

**Тренд:** индустрия движется от POC (прототипов) к продакшену. Вопрос уже не «стоит ли строить агентов», а «как внедрять их надёжно и в масштабе».

## 6. Что такое Harness (анатомия агентской обвязки)

**Agent = Model + Harness**

Если ты не модель — ты строишь harness.

**Harness** — это весь код, конфигурация и логика исполнения, которые не являются самой моделью. Модель сама по себе — не агент. Агентом её делает harness.

### Компоненты harness:
- **System prompts** — инструкции модели
- **Tools / Skills / MCP** — инструменты и их описания
- **Инфраструктура** — файловая система, песочница, браузер
- **Логика оркестрации** — спавн саб-агентов, передача задач, маршрутизация между моделями
- **Hooks / Middleware** — компакшн, продолжение работы, проверки

### Проблемы, которые решает harness:

**1. Контекст (context rot):** Модель тупеет когда контекст переполняется.
   — *Compaction:* суммаризация контекста
   — *Tool call offloading:* выгрузка больших выводов инструментов в ФС
   — *Skills:* загрузка навыков по мере необходимости

**2. Долгие задачи (long horizon):** Агент должен работать автономно долго.
   — Файловая система + git для сохранения прогресса
   — **Ralph Loop:** хук перехватывает попытку завершить работу и запускает заново с чистым контекстом, подтягивая файлы из прошлой итерации

**3. Безопасность:** Песочницы (sandboxes) для изолированного выполнения кода.

### Будущее harness:
- Часть функционала harness будет впитываться в модели (модели будут уметь планировать и работать долго «из коробки»)
- Но harness engineering останется важным — правильная обвязка делает любую модель эффективнее
- Пример: изменение только harness подняло кодинг-агента с Top 30 до Top 5 в Terminal Bench

## 7. Что такое Agent Engineering (определение из State of Agent Engineering)

> Agent engineering — это итеративный процесс превращения LLM в надёжные системы. Поскольку агенты недетерминированы, инженерам нужно быстро итерировать, чтобы улучшать качество агентов.
